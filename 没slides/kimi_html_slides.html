署名とは

デジタル署名は、メッセージの真正性と改竄検知を保証する仕組みです。

Alice
秘密鍵で署名
→
メッセージ
+ 署名
→
Bob
公開鍵で検証
仕組み:
1. Aliceが自分の秘密鍵でメッセージに署名
2. BobがAliceの公開鍵で署名を検証
3. 検証が通れば「Aliceが書いた」かつ「改竄されていない」ことが保証される

参考: デジタル署名 - Wikipedia

RSAとECCの最大の違い

安全性の根拠

RSA: 素因数分解の困難性

大きな合成数を素因数分解することは計算量的に困難

N = p × q

p, q: 大きな素数
N から p, q を求めるのは困難

RSAのアルゴリズム詳細は省略

ECC: 離散対数問題の困難性

素数 $p$ と、$p$ 未満の自然数 $g, x$ を用いて:

$$g^x \equiv m \pmod{p}$$
  • $m$ を求めるのは簡単（順に計算）
  • $g, p, m$ から $x$ を求めるのは困難
  • $x$ は $g$ を底とする $m$ の離散対数

ECC (楕円曲線暗号) とは

主に公開鍵暗号方式で用いられる、楕円曲線上の離散対数問題の困難性を利用した暗号の種類

楕円曲線と演算の定義

平面上での楕円曲線とは、以下のような形で表される平面曲線です:

Weierstrassの標準形

$$y^2 = x^3 + ax + b$$

ECDSAなどで使用

Edwards曲線

$$ax^2 + y^2 = 1 + dx^2y^2$$

EdDSAで使用

  • パラメータ ($a, b$ や $a, d$) の値はアルゴリズムごとに異なる
  • 曲線が非特異であることが必要（判別式 $\neq$ 0）
  • 実平面上に限らない一般的な楕円曲線はより複雑（本筋と関係ないため省略）

代表的な曲線例

secp256k1

  • ECDSAなどで使用
  • Bitcoinで使用されている
  • Weierstrass形式: $a=0, b=7$
$$y^2 = x^3 + 7$$

Ed25519

  • EdDSAの曲線
  • ssh-keygenでよく見るやつ
  • Edwards形式: $a=-1, d=-\frac{121665}{121666}$
$$-x^2 + y^2 = 1 - \frac{121665}{121666}x^2y^2$$
その他のパラメータ:
法となる素数 $p$、ベースポイント $G$、$G$ の位数 $n$ なども重要なパラメータ

ECDH: 楕円曲線Diffie-Hellman鍵交換

手順

  1. Alice, Bob間で曲線やベースポイント $G$、その位数 $n$ を共有
  2. Alice, Bobが互いにランダムな秘密の整数 $d_A, d_B \in [1, n-1]$ を定め、
    $Q_A = d_A G$、$Q_B = d_B G$ の点の座標だけ教え合う
  3. Aliceは $d_A Q_B$、Bobは $d_B Q_A$ を計算
    → これらは同じ値 $K = d_A d_B G$ になる
ポイント 1:
$d_A$ をAliceの秘密鍵、
$Q_A$ をAliceの公開鍵と呼ぶ
両者は自分の秘密鍵を明かすことなく、
同じ値 $K$ を共有した
ポイント 2:
通信が傍受されていても、
$d_A$ と $d_B$ のどちらも知らない第三者は
$K$ を作れない
$K$（楕円曲線上の点）のx座標を共通鍵の素とすれば、
信頼できない通信路上でも安全に二者間の共通鍵を作成できる

ECDSA: Alice (署名者)

Elliptic Curve Digital Signature Algorithm

SSL証明書やBitcoinの署名など、幅広く使われている楕円曲線暗号署名のスタンダード
1999年にNISTにより標準化

署名手順

  1. Aliceが秘密鍵 $d_A \in [1, n-1]$ を作り、公開鍵 $Q_A = d_A G$ を計算
  2. メッセージ $m$ のハッシュ値を $e$ とする ($e = H(m)$)
  3. 位数 $n$ のビット長分だけ $e$ の上位をマスクして、この値を $z$ とする
  4. 乱数 $k \in [1, n-1]$ を作って、$(x, y) = kG$ を計算
  5. $r = x \mod n$ と $s = k^{-1}(z + r d_A) \mod n$ を計算
  6. $(r, s)$ を署名として、メッセージ $m$ と一緒にBobに送信
$k$ はnonceと呼ばれ、内部的な秘密鍵となる。
$k$ は有限体 $\mathbb{F}_n$ の元なので、$k^{-1}$ が存在する。

ECDSA: Bob (検証者)

検証手順

  1. $(r, s)$ を受け取る
  2. Aliceと同様の手順でメッセージ $m$ のハッシュ値 $e$ と、そのマスク $z$ を計算
  3. $u_1 = z s^{-1} \mod n$、$u_2 = r s^{-1} \mod n$ を計算
  4. $(x', y') = u_1 G + u_2 Q_A$ を計算
  5. $x' \equiv r \equiv x \pmod{n}$ なら、正当な署名
公開情報:
パラメータ ($p, G, n$)、Aliceの公開鍵 $Q_A$、
メッセージ $m$、署名 $(r, s)$
手計算で実際に変数を展開していくと検証される様子がわかって気持ち良い

余談: PS3事件

固定nonceの悲劇

SONYのミス

プレステ3のソフトウェア署名にECDSAを採用していたが、
全ての署名で同じnonce $k$を使い回してしまった

2010年

fail0verflowチーム (marcan, bushing, sven) と
George Hotz が脆弱性から署名用秘密鍵 $d_A$ を逆算することに成功

結果

jailbreak（脱獄）が可能に → 非公式ゲーム実行可能に

なぜ秘密鍵が漏れるのか:
同じ$k$で2つのメッセージに署名すると、連立方程式から$k$と$d_A$が解ける
$s_1 = k^{-1}(z_1 + r d_A) \mod n$
$s_2 = k^{-1}(z_2 + r d_A) \mod n$
$k \equiv \frac{z_1 - z_2}{s_1 - s_2} \mod n$

余談2: Snowden事件

楕円曲線暗号に潜むバックドアの可能性

NIST-p256などの楕円曲線の乱数生成器としてNSAが推奨していた
Dual_EC_DRBG に、バックドアが仕込まれていたのではないかという疑惑

問題の所在

  • Dual_EC_DRBGの安全性は、2つのベースポイントの離散対数問題に依存
  • そのベースポイントの決定根拠が不明瞭
  • NSAが事前に解となるスカラー値を把握しているポイントを
    あえて設定したのではないか、という疑念
影響:
乱数生成の生成パターンが予想可能になると、
nonceなどの値が割れてしまい、結果的に秘密鍵の復元に使われかねない

参考: 楕円曲線暗号の標準化とバックドア

Schnorr署名

1989年にClaus P. Schnorrによって発明

2008年まで特許の都合であまり利用されていなかったが、
ECDSAより検証の効率が良く、集約署名が可能
Alice (署名者)
  1. パラメータ共有 ($G$, $n$)
  2. 秘密鍵 $d_A$ と公開鍵 $Q_A = d_A G$ を用意
  3. 乱数 $r$ を用いて $R = rG$ を計算
  4. メッセージ $m$ と $R$ を結合 ($R \| m$)
  5. $s = r + H(R \| m) d_A$ を計算
  6. $(R, s)$ を署名として送信
Bob (検証者)
  1. $(R, s)$ を受け取る
  2. $sG = R + H(R \| m) Q_A$ が成り立てば正当
証明:
$sG = (r + H(R\|m)d_A)G$
$= rG + H(R\|m)d_AG$
$= R + H(R\|m)Q_A$ ✓
めちゃくちゃシンプルです

EdDSA

Edwards-curve Digital Signature Algorithm

注意: 今までのWeierstrass形式の曲線ではなく、
twisted Edwards曲線を用いている
Alice (署名者)
  1. パラメータ共有 ($G$)
  2. 秘密鍵 $d_A$ のハッシュ $H(d_A)$ の前半32バイトを $s$ とし、
    公開鍵 $Q_A = sG$ を計算
  3. $H(d_A)$ の後半32バイトを $\text{prefix}$ とする
  4. $r = H(\text{prefix} \| m) \mod n$ として、$R = rG$ を計算
  5. $k = H(R \| Q_A \| m) \mod n$ を計算
  6. $S = (r + ks) \mod n$ として、$(R, S)$ を送信
Bob (検証者)
  1. $(R, S)$ を受け取る
  2. $k = H(R \| Q_A \| m) \mod n$ を計算
  3. $8(SG) = 8(R + kQ_A)$ が成立すれば正当
Schnorrとの違い:
  • nonce $r$ を決定論的に計算
    = 乱数生成器を用いていない
  • 8 (cofactor) が登場する

EdDSAの特徴

決定論的nonce

秘密鍵のハッシュから決定論的に計算される

メリット:
乱数生成器の安全性に依存しない

PS3やSnowdenの問題を回避

cofactor 8

twisted Edwards曲線では位数が $n = 8L$ (8の倍数)

完全加法公式:
点による場合分けが不要 →
サイドチャネル攻撃への耐性

小位数部分群の影響をキャンセルするため8をかける

twisted Edwards形式の加法公式

$$x_3 = \frac{x_1 y_2 + x_2 y_1}{1 + d x_1 x_2 y_1 y_2}, \quad y_3 = \frac{y_1 y_2 - a x_1 x_2}{1 - d x_1 x_2 y_1 y_2}$$

$P = Q$ の場合も同じ式が使える（完全加法公式）

応用例: SSH

1. 鍵交換
ECDHEなどで共有秘密を作成 → 共通鍵を導出
↓
2. サーバ認証
サーバはホスト鍵で自分を証明 → クライアントは~/.ssh/known_hostsと照合 or 登録
↓
3. ユーザ認証
サーバがチャレンジを送信 → クライアントは秘密鍵で署名 → 公開鍵で検証
楕円曲線暗号の使用箇所:
  • 鍵交換: curve25519-sha256 など
  • ホスト鍵・ユーザ認証: ssh-ed25519, ecdsa-sha2-nistp256 など

応用例: Bitcoin

1. UTXOをアンロック

未使用の自分宛送金トランザクション(UTXO)を秘密鍵で署名して使用可能に

2. トランザクション作成

署名と公開鍵つきUTXOをinput、相手の公開鍵ハッシュをoutputとして登録

3. 検証

マイナーが公開鍵ハッシュ計算と署名検証を行い、正当ならプールに保持

4. ブロック化 (PoW)

マイナーがトランザクションを集めてブロックを作成、nonce探索
